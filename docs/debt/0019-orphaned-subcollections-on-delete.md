## Context

When a user deletes a pet from the web UI, the frontend client executes a batched write to delete the main document in `pets/{petId}` and the emergency profile in `emergency_profiles/{petId}`. However, Firestore does not cascade deletes to subcollections automatically. The subcollections generated by the AI agent (`health_events`, `reminders`, `grooming_records`, and `conversations`) remain orphaned in the database.

## Impact

Over time, this will accumulate dead data, consuming storage quotas (though negligible for a POC). Furthermore, if a new pet were ever assigned the same UUID (astronomically unlikely) or if we run collection group queries, the orphaned data would resurface. For regulatory compliance (e.g. GDPR/Ley 29733 right to be forgotten), leaving orphaned identifiable medical history is problematic long-term.

## Current workaround

None — tolerated for the POC phase. The frontend simply removes the main pointer documents (`pets`, `emergency_profiles`), so the pet disappears from all user interfaces and public URLs. 

## Proposed fix

Implement a Cloud Function triggered by `onDocumentDeleted('pets/{petId}')`. This function should use the Admin SDK to recursively list and delete all documents within the known subcollections (`health_events`, `reminders`, `grooming_records`, `conversations`) and any other related artifacts.
